- (decision) Why: Solve the problem of project managers allocating work to specialists while enforcing capacity limits, prerequisites, and recovery floors, ensuring a fair and sustainable workload in a local-first, zero-config environment.
- (capability) CAP-1: Specialist Management. Intent: PMs can create, edit, and deactivate specialists with skill, seniority, and workload profiles. Success: Specialist records are persisted and retrievable.
- (capability) CAP-2: Task Management. Intent: PMs can create, edit, and manage dependency links between tasks. Success: Tasks with valid durations and dependencies are persisted.
- (capability) CAP-3: Allocation Management. Intent: PMs can assign specialists to tasks over a time window. Success: Assignments are blocked if they violate capacity or dependency rules.
- (capability) CAP-4: Override Management. Intent: PMs can explicitly override constraints for urgent assignments. Success: Overrides capture priority, rationale, and create capacity debt records.
- (capability) CAP-5: Audit and Monitoring. Intent: System maintains an immutable history of all changes and overrides. Success: Audit events are append-only and traceable.
- (capability) CAP-6: Personal Health Shield. Intent: Specialists receive detailed utilization heatmaps and fatigue alerts. Success: View reflects individual utilization and flags fatigue based on rules.
- (capability) CAP-7: Risk Intelligence. Intent: System flags schedule risks where dependencies slip beyond task start dates. Success: Risk indicators appear on the dashboard for affected tasks.
- (capability) CAP-8: Privacy-Aware Views. Intent: Role-based boundaries restrict PMs to aggregate status and specialists to detailed personal data. Success: Data access layer enforces role-based filtering.
- (constraint) Local SQLite persistence.
- (constraint) Pure TypeScript logic for constraint engines.
- (constraint) Server-driven state updates (Server Actions).
- (constraint) Local-first, zero-config (no auth/sync complexity).
- (constraint) Role-aware data boundaries enforced in the data access layer.
- (constraint) Task duration minimum: 0.25 hours.
- (constraint) Prerequisites must be complete before task start.
- (constraint) Specialist load ≤ 100% unless overridden.
- (constraint) Recovery floor rules enforced for intensive tasks.
- (decision) Non-goals: Cloud synchronization, multi-user collaboration, and complex authentication systems.
- (decision) Success signal: A PM can successfully allocate a set of tasks to specialists, with the system correctly blocking invalid allocations and recording necessary overrides, while specialists can view their own fatigue alerts without the PM seeing their private health data.
