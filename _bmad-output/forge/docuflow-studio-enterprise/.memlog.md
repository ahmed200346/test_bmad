- (lock) Scope is single-user/local test; Enterprise refers to feature complexity, not deployment scale. Local JSON storage is locked.
- (assumption) Prototype will be tested by multiple users (colleagues, professors) in a non-production environment, contradicting the 'single-user' lock but maintaining JSON for the prototype phase.
- (decision) Abandoned local JSON storage in favor of a real database (MySQL or simple local DB) to support multi-user testing by colleagues/professors.
- (lock) Storage locked to SQLite for the prototype phase to ensure zero-config portability while maintaining SQL integrity and concurrency. Migration path to MySQL/Postgres for production.
- (lock) Integrity rules are developer-defined (hardcoded/schema-driven), not user-configurable.
- (note) User is acting as both Legal and Developer; this collapses the organizational separation of concerns for the prototype phase.
- (lock) Integrity rules are stored as data in SQLite (Rule Manifest), not hardcoded logic, to ensure adaptability and separation of rules from the execution engine.
- (lock) Dependency resolution implemented as 'Tiered Execution' (integer-based levels in SQLite). Simple, linear resolution suitable for academic prototype.
- (lock) Locked: Visual Diff Viewer and Full Version History. These are required for the academic demo despite the implementation overhead.
- (lock) Versioning implemented via Snapshot Pattern (full JSON blobs in contract_versions table). Diff Viewer will target the rendered final document to maintain 'Enterprise' ergonomics, falling back to JSON diff only if rendering is unfeasible.
